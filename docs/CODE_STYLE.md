# 代码规范

## 通用原则

- 改动保持聚焦，避免无关重构。
- 优先沿用项目已有结构和命名。
- 配置、模板和工具定义优先使用结构化 JSON，不使用零散硬编码。
- 不提交日志、IDE 文件、构建产物和任何密钥。

## Java

- 公共类和非平凡公共方法需要写 Javadoc。
- Javadoc 描述当前行为、输入输出和安全约束。
- 不写变更历史类注释，例如日期、聊天记录、临时决策、用户原话。
- 不在代码注释中引用 Issue 争论或聊天上下文。
- 行内注释只解释不容易从代码本身看出的逻辑。
- Provider 包名保持清晰，例如：
  - `provider/aliyun/oss`
  - `provider/aliyun/dns`
  - `provider/db`
  - `provider/nosql/redis`
  - `provider/ops/ssh`

## Vue / TypeScript

- 组件注释只保留维护者真正需要的信息。
- 不保留模板项目的示例页面和无业务入口组件。
- API 调用放到 `src/api` 下，保持类型清晰。
- 协议差异字段使用数据驱动配置渲染，避免页面里写大量分支。
- UI 优先使用 Art design pro 和项目已有组件。

## SQL

- 新增表结构变更放到 `mcp-gateway/src/main/resources/db`。
- SQL 尽量可重复执行。
- 破坏性迁移需要在 PR 中明确说明影响。

## 日志

- 应用日志不能输出明文密码、Token、私钥、TOTP、云密钥或带凭证的连接串。
- 调用日志可以为了审计保存入参和结果，但展示和落库前必须经过敏感字段隐藏策略。
